# Sessão 2026-03-18 — Trajetória Completa

## O que aconteceu

Sessão longa que começou recuperando contexto de um crash do VSCode e terminou redefinindo a identidade do projeto. Quatro fases que construíram uma em cima da outra.

---

## Fase 1: Recovery + Coverage Gaps

A sessão anterior travou no meio de um plano de coverage. Recuperamos o contexto pelos logs do Claude, achamos 92 linhas de testes não commitados e um pedido pendente de "rodar coverage + integração real".

Fechamos os gaps: 10 testes novos cobrindo timeout 504, exception 400, encoding UTF-8/latin-1, config 503, pipeline timeout, transcribe domain error, process Document extraction, health root, engine dependency cycle e None result.

Resultado: módulos API (config, health, process, transcribe, analyze) todos em 100% coverage. Engine em 96%. De 762 pra 772 testes.

O Codex rodou uma análise e pegou 3 achados reais: `__main__.py` quebrado (corrigido), testes de visualização falhando em sandbox (ambiental, não bug), e warnings de `to_thread` never awaited (artefato de mock, filtrado). Saída do pytest ficou limpa: zero warnings.

## Fase 2: Visualizer Rendering Refactor

A análise do Codex sobre kaleido e bordas de renderização levou à pergunta: "o que vai facilitar a criação de plugins?". A resposta: o plugin não deveria se preocupar com serialização.

Refator completo: plugins passaram a retornar o objeto de figura nativo (plotly.Figure, matplotlib.Figure, HTML string) e o BaseVisualizerPlugin._serialize() detecta o tipo via duck-typing e serializa pro formato que o consumer pediu. Formatos disponíveis detectados dinamicamente (kaleido opcional).

Três plugins renomeados pra refletir a lib: wordcloud_viz → wordcloud_d3, frequency_chart → frequency_chart_plotly, sentiment_viz → sentiment_viz_plotly. O sentiment_viz teve o chart_type `distribution` convertido de matplotlib pra plotly — eliminou ambiguidade de lib mista.

Rota `/visualize` simplificada: sem temp files, sem FileResponse, sem BackgroundTask. Frontend: default PNG → HTML. Template `create_plugin.py` atualizado.

13 arquivos de teste atualizados, 772 testes mantidos.

## Fase 3: Mapeamento do Ecossistema + Ecosystem Readiness

Pergunta: "quero entender o quadro completo dos meus projetos". Exploramos 5 projetos em paralelo:

- **DeepVoC** — pipeline de pesquisa qualitativa VoC com embeddings, clustering, BERTopic, RoBERTa, automação via Claude Code CLI
- **Observatório VoC** — operação de surveys Qualtrics (funis, waves, bounces, power analysis) com Polars
- **whats-le** — análise de chat WhatsApp (44k mensagens, parsing, enrichment, sentimento, emoji)
- **transcript-analyser** — predecessor direto do Qualia (9 analyzers, 8 charts, ConfigurationRegistry)
- **DataLab** — sidebar React pro Google Sheets (preparação de dados de pesquisa)

Resultado: ECOSYSTEM_MAP.md com ~30 plugins candidatos organizados por camada (NLP pesada, linguística, document processing, visualização).

A discussão revelou que o Qualia é puxado em duas direções: NLP/ML pesado (DeepVoC) e análise leve (Observatório, whats-le). Decisão: absorver ambas, mas organicamente — plugins vêm quando cada projeto precisar.

Duas mudanças no engine:
1. **Provides relaxado** — `_provides_map` mudou de `Dict[str, str]` pra `Dict[str, List[str]]`. Múltiplos plugins podem declarar o mesmo campo. O consumer escolhe qual rodar.
2. **Extra `[ml]`** — PyTorch, transformers, sentence-transformers, scikit-learn, umap-learn.

De 772 pra 776 testes.

## Fase 4: Identidade — README v3 + Nome + Conceito

A conversa sobre o ecossistema levou naturalmente a: "o que o Qualia realmente é?". A resposta: não é um motor de análise qualitativa. É um **workbench de análise local-first** — uma bancada onde transformações de dados acontecem via plugins configuráveis, acessível por qualquer ferramenta, sem mandar dados pra fora.

README reescrito em 3 versões iterativas com o usuário:
- v1 (atual na época): tom bom mas posicionamento desatualizado ("framework bare metal", "core burro")
- v2: estrutura nova mas sem alma ("parece documentação")
- v3 (final): melhor dos dois mundos — abertura com o conceito de workbench local-first, explicação de plugins com tipos e libs, DX de criação de plugin, extras modulares

Repo renomeado de `mrlnlms/qualia` pra **`mrlnlms/qualia-core`** — posicionando como motor central do ecossistema Qualia (qualia-core, qualia-coding, futuro qualia-gsheets).

A exploração do Foundations.md (documento de 326KB sobre Mixed Methods) revelou a conexão mais profunda: o Qualia materializa o conceito de **transformação de dados** da literatura de métodos mistos. Quantitização (texto → números) e qualitização (números → narrativas) são o ciclo central da pesquisa — e cada plugin do Qualia é uma materialização de um passo desse ciclo. Isso foi incorporado no README como um parágrafo no "Por que":

> O trabalho do pesquisador é, no fundo, transformar dados — texto vira código, código vira frequência, frequência vira cluster, cluster vira perfil narrativo. Cada passo é uma decisão interpretativa. Qualia torna essas transformações explícitas, configuráveis e reutilizáveis.

## Análise Final

O Qualia saiu dessa sessão numa posição diferente de onde entrou.

Entrou como "motor de análise qualitativa com 8 plugins e boa cobertura de testes". Saiu como o centro de um ecossistema mapeado, com arquitetura pronta pra escalar e identidade definida.

A mudança mais importante não foi técnica — foi conceitual. O visualizer refactor e o provides relaxado são boas engenharias, mas o que realmente mudou foi entender o que o projeto é: uma bancada de transformação de dados onde cada plugin materializa uma decisão interpretativa do pesquisador. Isso veio da conversa, não do código.

Tecnicamente, o engine tá num ponto onde não tem mais o que polir. 776 testes, rotas da API em 100% coverage, interfaces limpas, serialização resolvida, dependências topológicas, cache, plugins coexistentes. A infra não é mais gargalo — o gargalo agora é conteúdo (plugins) e comunicação (fazer a visão ser entendida).

O que ficou pra frente:
- Plugins vêm organicamente dos projetos reais (ECOSYSTEM_MAP como catálogo)
- A visão profunda (Foundations.md ↔ código) ainda não tá acessível publicamente além de um parágrafo no README
- O frontend existe mas não foi tocado — quando os plugins pesados chegarem, o Workflow vai ser o lugar onde o pesquisador monta as transformações visualmente

A sessão foi longa mas cada peça construiu em cima da anterior: coverage → refactor → ecosystem → provides → README → identidade. Não foi dispersão — foi uma espiral que convergiu.
